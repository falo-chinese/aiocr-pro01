Pro01 Viewer Prompt Lab Runtime Notes

Last updated: 2026-05-18

定位

Pro01 是 standalone / GitHub Pages friendly 的單檔 HTML runtime，不依賴 Pro02 OCR server。它不是單純發票 OCR 頁，而是泛用文件 AI ETL / HITL workbench。

image / document / form
-> template field schema
-> prompt schema
-> AI1 / AI2 proposal
-> human reconciliation
-> evidence package export

tw_invoice_current_fields_v1

8 欄：invoice_number, date, seller_tax_id, buyer_tax_id, item_description, amount, tax_amount, total_amount。

台灣統一發票（三聯式）－精簡版

Tab 設計

Tab 1 主工作區：選模板、載入圖片、執行 A1/A2、人工確認、匯出。

Tab 2 模板範例：查看欄位 schema、核心 JSON、Prompt，並套用到主工作區。

Tab 3 系統參數：目前先管理送出圖片策略。

AI 執行

  • 執行到 A1
  • 執行到 A2
  • 雙模型執行：依序跑 A1 gemini-2.5-flash-lite，再跑 A2 gemini-2.5-flash

按鈕狀態：黃色代表執行中，綠色代表成功，紅色代表失敗。

Pass 欄位

每個 reconciliation row 可以採用 AI1、採用 AI2，或 Pass 此欄位。Pass 不會讓資料默默消失，而是留下人工略過紀錄。

{
  "validation_status": "passed",
  "pass_status": true,
  "pass_note": "人工略過此欄位",
  "confirmed_relation": "人工 Pass"
}

送出圖片策略

轉檔由瀏覽器 Canvas 在本機完成，不需要 server。預設是原檔。

策略格式尺寸規則備註
original原格式不轉檔evidence-first default
jpg_compressedJPEG長邊超過 1400 才縮小小檔案、速度快
jpg_highJPEG長邊超過 2200 才縮小照片型文件較適合
png_standardPNG長邊超過 1800 才縮小推薦 OCR / AI vision
png_highPNG長邊超過 3000 才縮小小字票據，檔案較大

重要決策：不放大小圖。小圖放大不會創造真實細節，反而增加 token / 傳輸成本。

圖片 Metadata

runtime 會記錄 source image 與 actual sent image 的格式、大小、寬高、轉檔策略、是否 resize、scale、quality 等資訊。

{
  "source_image_meta": {
    "name": "",
    "type": "",
    "size": 0,
    "size_kb": 0,
    "width": 0,
    "height": 0
  },
  "last_send_image_meta": {
    "name": "",
    "type": "",
    "size": 0,
    "size_kb": 0,
    "width": 0,
    "height": 0,
    "strategy": "",
    "converted": true,
    "resized": false,
    "scale": 1
  }
}

重要決策：本機 DB 與 JSON / Excel 匯出不保存圖片位元組，只保存圖片 metadata。這樣可以降低瀏覽器容量壓力，也方便之後做雲端上傳分析時避免把原始圖片一起送出。

本機儲存閉環

Pro01 的本機版應該不用 server 就能完成一個閉環：目前工作區 → 自動記憶到瀏覽器 IndexedDB → 匯出 JSON / Excel → 匯入 JSON → 後續做地端或雲端彙總分析。

使用者不需要一直按「儲存記憶」。A1/A2 更新、模板切換、欄位確認、人工修改、Pass 欄位、圖片策略調整，都應該由 runtime 自動記憶。

再次開啟頁面時，可載入最近一筆本機紀錄回工作區。不過因為不存圖片，所以只恢復文字 proposal、欄位、模型模式、圖片 metadata 與分析 context。

畫面上真正會調整專案記憶空間的管理動作只保留「清除 Pro01 記憶」。匯出 / 匯入是資料交換能力，不是日常記憶按鈕。

層次用途是否存圖片說明
localStorage小型設定否API Key、圖片策略、A1/A2 模式、Prompt 記憶
IndexedDB使用者暫存紀錄否保存 reconciliation payload、欄位、metadata、analysis seed
JSON 匯出可攜備份 / 雲端分析否最適合程式讀取與批次分析
Excel 匯出人工檢視 / 教學否以表格方式攤平紀錄
HTML 匯出可讀 evidence package機器 JSON 不存圖片分析器應讀 script blocks，不應抓畫面排版

清除專案記憶時，只清 Pro01 的 localStorage keys 與 IndexedDB records，不應清其他專案、瀏覽器快取或系統模型。

Portable Analysis Package

未來不論是地端掃描資料夾裡的 HTML/JSON，或雲端上傳幾個 evidence files 做分析，都應該讀取穩定的 machine-readable JSON blocks。

新的 canonical script block ids：

pro01-package-manifest
pro01-record
pro01-analysis-seed

舊版相容 blocks 可保留：

falo-runtime-payload
falo-confirmed-result
falo-reconciliation-fields
falo-accuracy-analysis-seed
falo-audit-summary

pro01-package-manifest 用來快速判斷檔案是不是 Pro01 package；pro01-record 保存完整紀錄但不含 API Key、不含 base64 圖片；pro01-analysis-seed 保存 AI1 vs AI2 vs Human Confirmed 的正確率分析種子。

{
  "package_type": "falo_pro01_reconciliation_package",
  "schema_version": "1.02",
  "runtime": "pro01_viewer_prompt_lab",
  "record_id": "pro01_...",
  "contains_image_bytes": false
}

這樣地端與雲端可以共用同一套解析邏輯：

local folder scanner:
scan *.html / *.json -> extract pro01-* blocks -> aggregate

cloud upload:
upload selected *.html / *.json -> extract pro01-* blocks -> aggregate

Accuracy Analysis Seed

開發模式先記錄詳細資料。正確率分析比較 AI1、AI2 與 Human Confirmed，其中 Human Confirmed 暫時視為 ground truth。

分析 JSON 會記錄模板、圖片策略、原圖/送出圖 metadata，以及 AI1 / AI2 的模式與模型。

{
  "analysis_context": {
    "analysis_basis": "human_confirmed_as_ground_truth",
    "template": {
      "id": "tw_einvoice_a_full_v1",
      "name": "電子發票 A－完整版"
    },
    "image_strategy": {
      "mode": "png_standard",
      "label": "一般 PNG（推薦）"
    },
    "models": {
      "ai1": {
        "source": "gemini_api",
        "model": "gemini-2.5-flash-lite"
      },
      "ai2": {
        "source": "gemini_api",
        "model": "gemini-2.5-flash"
      }
    }
  }
}

每個欄位也會記錄 value + source + model、是否與 confirmed 相符、AI1/AI2 是否一致、是否 pass。

匯出的 HTML 會包含 falo-accuracy-analysis-seed JSON script block，方便後續批次分析。
